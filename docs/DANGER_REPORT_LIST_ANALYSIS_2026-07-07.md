# 「報告一覧」(`/report`) 分析レポート

- 対象: `app/report/page.tsx`(ナビゲーション上のラベルは「報告一覧」)
- 実施日: 2026-07-07
- ブランチ: `codex/danger-report-region-filter-stability`
- 手法: 関連コードの読み取り + Supabase `danger_reports` の読み取り専用集計(service role、書き込みなし)

## 1. アプリコンセプトの総評

PathGuardianは「子どもが危険を"見つける遊び"を通じて学び、その成果を家族と学校が共有する」という一貫した物語を持っており、たんけんノートのデザイン統一とルペのキャラクター設計はその物語をよく支えている。危険報告→地図ピン→家族共有カード→教育版PDFという「1つの報告が複数の形で価値を持つ」パイプラインは、このジャンルのアプリとして構造的に優れている。ただし「報告一覧」ページはこのパイプラインの中で位置づけが最も曖昧で、ナビ上のラベル(報告一覧)・内部説明(危険報告の統計)・ページ見出し(みんなの危険報告)の3つが食い違っており、SNS的フィードと安全情報アーカイブのどちらを目指すのか設計意図が定まっていない。さらに実データを見ると、公開18件の全件で `prefecture` が null のため、本ブランチで実装した地域フィルタの価値がこのページには届かない状態である。

## 2. 報告一覧ページの強み(根拠つき)

- **画像プライバシーの一貫した扱い**: 署名URLマップを1回だけ構築して全表示箇所(カテゴリカード/フィード/詳細シート)で使い回す設計(`app/report/page.tsx:186-203`)。非公開バケット維持の方針(`lib/danger-report-image-access.ts:6-20` のコメント)とも整合。
- **家族共有カードの動線が詳細シートに内蔵**: 詳細を見た流れのまま `shareFamilyShareCard` で Web Share API→ダウンロード+文面コピーへフォールバック(`lib/report-generation/family-share-card.ts:161-191`)。「見る→家族に伝える」の距離が短い。
- **要約・行動のフォールバックが空データに強い**: description が空でもタイトルから要約文を合成(`family-share-card.ts:53-68`)、確認アクションも既定文を保証(`:70-84`)。実データで description 空が4/18件ある現状に効いている。
- **インタラクションのバッチ取得**: 60件分のいいね/保存を `useReportInteractionsBatch` で一括取得し楽観的更新+ロールバック(`hooks/use-report-interactions.ts:310-541`)。N+1を避けた実装。
- **未ログイン時の分かりやすいゲート**: いいね/保存はトーストで「ログインが必要です」(`use-report-interactions.ts:411-418`)、コメント欄はログインリンク付きプロンプト(`report-comment-section.tsx:176-189`)。閲覧自体はブロックしない。

## 4. 改善提案(優先度別)

### A. 今すぐ小さく直せるもの

**A-1. prefecture のバックフィル(このブランチの地域フィルタを機能させる前提条件)**
- なぜ: 新規投稿は `reverseGeocodeLocation` で prefecture を保存する(`hooks/use-danger-report-submit.ts:100-115`)が、既存公開18件は全件 null。landing の地域フィルタは `query.eq("prefecture", selectedPrefecture)`(`components/landing/HiyariHatReport.tsx:126`)なので、**都道府県を選んだ瞬間に既存データが全て消えて0件表示になる**。「関係ない地域ばかりで離脱」を直すはずの機能が「何もない地域」体験を作る。
- どこ: 全18件に座標はある(noCoords=0)ので、`lib/map/reverse-geocode.ts` を使った一回きりのバックフィルスクリプトで解決可能。
- 影響範囲: DBのみ。コード変更不要。

**A-2. `formatDate` に年を追加**
- なぜ: `app/report/page.tsx:92-104` は `month: "short", day: "numeric"` のみで年を出さない。実データは2025年4月〜2026年7月の15ヶ月に跨っており、**1年前の「4月29日」の災害報告が今年のものに見える**。安全情報で鮮度の誤認は実害になる。
- どこ: `formatDate` に `year` を追加するか、「○ヶ月前」の相対表記+古い報告への「情報が古い可能性」バッジ。
- 影響範囲: このページ内の1関数。

**A-3. ページの名前を1つに統一**
- なぜ: ナビラベル「報告一覧」(`components/ui/navigation.tsx:146`)・description「危険報告の統計」(`:148`)・ページ見出し「みんなの危険報告」(`page.tsx:489`)の3つが不一致。統計は実際には件数サマリー1行しかなく、「統計」という説明は誤り。
- どこ: `navigation.tsx:142-150` の `secondaryNavItems`。「みんなの危険報告」への統一を推奨。
- 影響範囲: ラベル文字列のみ。

**A-4. 「気をつけよう」ボタンの永続化 or 削除**
- なぜ: `cautionStates` はローカル state のみ(`page.tsx:111, 319-324`)で、リロードで消え、DBにも残らず、押しても何も起きない見た目だけの機能。「AIスロップに見える」指摘で偽コンテンツを排除した方針([[parent-first-ui-overhaul-2026-07-04]])と矛盾する"偽インタラクション"。
- どこ: `page.tsx:462-470`。永続化するなら `report_likes` と同型のテーブルが必要なので、短期では削除が正しい。
- 影響範囲: このページのみ。

### B. 中規模の設計変更

**B-1. 報告一覧にも地域フィルタを接続する**
- なぜ: このブランチで `lib/user-region.ts` を唯一の真実源として landing と地図に都道府県フィルタを接続済み(コミット `b503f4bd3` / `7da27ba33`)だが、**報告一覧ページだけ未接続**。取得は「全国・最新60件・降順」固定(`page.tsx:129-136`)。ユーザーが landing で地域を選んでも、報告一覧に来ると無関係な地域の報告が混ざる。
- どこ: `HiyariHatReport.tsx` と同じパターンで `getStoredRegion()` + 地域チップ + `eq("prefecture", ...)` を `page.tsx` の `fetchReports` に追加。A-1(バックフィル)が前提。
- 影響範囲: `app/report/page.tsx` + テスト。既存の `tests/components/hiyari-hat-report.test.tsx` が参考実装。

**B-2. 2タブ構成の再設計(カテゴリギャラリーの縮小)**
- なぜ: 「危険報告」タブと「共有フィード」タブは同じ60件を並べ替えただけで、カテゴリカードをタップして開くシートの中身も `renderShareCard` で**フィードと完全に同一**(`page.tsx:640-644`)。実データではカテゴリは実質2種(disaster 12・traffic 5)しかなく、5枚中3枚のカテゴリカードは0件か1件。
- どこ: タブを廃止しフィード一本+カテゴリチップ(フィルタ)に統合するのが第一候補。カテゴリの「最終更新」情報はチップ内バッジで代替可能。
- 影響範囲: `page.tsx` の大部分。ただし削る方向の変更なのでリスクは低い。

**B-3. SNS的機能の意味づけを「安全行動」に寄せる**
- なぜ: 「いいね」は安全情報に対して感情の向きが曖昧(危険報告に"いいね"？)。一方でアプリには家族共有・確認済みという固有の行動語彙が既にある。エンタメ化のリスクは機能の存在ではなく**ラベリング**にある。
- どこ: 「いいね」→「たすかった」等の感謝系ラベルへ変更(`page.tsx:446`)、「保存」はそのまま(実用的)。バックエンド(`report_likes`)は変更不要でラベルとアイコンのみ。
- 影響範囲: 文言+アイコン。テストの文言アサーション更新。

**B-4. 未ログイン閲覧時の画像表示の検証(未検証・要確認)**
- なぜ: storage の SELECT ポリシーは `TO authenticated` に変更済み(`danger-report-image-access.ts:13-14` のコメント)。`createSignedUrl` は anon クライアントで呼ばれた場合に失敗する可能性が高く、その場合**未ログインユーザーには全報告の画像が出ずグラデーションのプレースホルダだけが並ぶ**。これは新規ユーザーの第一印象=登録動機を直撃する。
- どこ: まず未ログイン状態での実機確認。画像が出ないなら、サムネイルのみ公開バケット/エッジで署名するAPI等の設計判断が必要。
- 影響範囲: 未検証のため確認が先。`danger_reports` テーブル自体の anon RLS も併せて確認すべき。

### C. 将来検討

**C-1. description の公開前モデレーション**: 不審者アラートはサーバ側LLM審査へ移行済みだが、通常の危険報告 description は `page.tsx:421-422` で生のまま表示。approved になる前に人間(管理者)が見る運用は byStatus(pending 3件)から機能していると推測できるが、学校名・個人名の混入を機械的に検知する層はない。投稿量が増えたら不審者アラートと同じ審査パイプラインに載せる。

**C-2. 鮮度・解決状態のライフサイクル表示**: resolved が2件あるがフィード上で approved と区別されない。「解決済み」バッジと、一定期間更新のない報告の「再確認募集」への転換は、データが古くなり続ける(直近90日で4件しか増えていない)この規模のサービスでは、コンテンツを死蔵させない重要な仕組みになる。

**C-3. 60件上限のページネーション**: 現状21件なので実害ゼロ。地域フィルタ実装後、1地域で60件を超えるまで着手不要。

## 5. これは提案しない(既存改善と重複のため除外)

- 危険レベル表示の一元化・4段階の色: `lib/report-generation/danger-level-presentation.ts` で対応済み。ただし本ページの `DANGER_TYPE_META`(`page.tsx:47-73`)はレベルでなく**タイプ**の配色でありそちらとは別物、と確認済み。
- 画像の署名URL化・非公開バケット維持: 対応済み(B-4は「未ログイン時の抜け」という別論点)。
- 地図ピンの採番・重なり・レポートPDF・「おうちのかたへ」文言: すべて対応済みのため触れない。
- 偽コンテンツ排除: 対応済み。ただしA-4「気をつけよう」は同じ思想の適用漏れとして新規指摘。

## 最重要の1点

このブランチの地域フィルタは実装として存在するが、**prefecture バックフィル(A-1)なしでは既存データに対して常に0件を返す**。マージ前にバックフィルを済ませるか、フィルタ側に「prefecture null は座標から都道府県を推定」のフォールバックを入れるかの判断が必要。
