# 親子で通学路確認ARモード 仕様書 v2.0

> 本書は v1.0（2026-05-03 初版）を、4 役レビュー（スタートアップCTO / SIer部長 / QAエンジニア / シニアエンジニア）の指摘を反映して全面改訂したものである。改訂履歴は章 13 を参照。

---

## 1. 概要

既存の AR 危険箇所ビューに、親子で通学路を歩きながら安全を学べる「親子で通学路確認」モードを追加する。

このモードでは、登録済み通学路の近くにある危険ポイントだけを対象にし、現地で近づいたタイミングで子ども向けの注意を表示する。歩行後にはチェックリストとクイズで理解度を確認し、結果をログインユーザーの学習履歴として保存する。

目的は、保護者が危険箇所を確認するだけでなく、子ども自身が「どこで止まるか」「何を見るか」「なぜ危ないか」を理解して行動できる体験にすること。

## 2. 背景と理由

通学路の安全対策では、保護者が危険箇所を把握するだけでは不十分である。実際に歩く子ども自身が、見通しの悪い交差点、車から見えにくい場所、歩道が狭い場所などを理解し、その場で適切に行動できることが重要になる。

単なる地図アプリではなく、親子で通学路を確認しながら学べる体験にすることで、継続利用、家庭内での安全確認、安全意識の向上につなげる。

## 3. 段階リリース方針

レビューで「現状仕様のまま全部実装するな」「PMF 検証前に DB を切るな」「法務未対応で出すな」が一致したため、**v1 MVP → v2 拡張** の 2 段階で進める。

### 3.1 v1 MVP（8 営業日、検証目的）

- 親子モードの体験コア（接近通知＋確認ボタン）と計測イベントだけ実装
- セッション保存は **IndexedDB のみ**（Supabase テーブルは作らない）
- AR 内クイズは作らず、既存 `/route-quiz?routeId=xxx` への誘導ボタンのみ
- 法務（規約・PP 改訂、AR 起因事故の警告 UI、保護者同意フロー）は v1 と並行で着手し、v1 リリース時に必須

### 3.2 v2 拡張（5 週間、v1 利用率検証後）

判定基準: v1 リリース後 30 日で `ar_parent_mode_started` の DAU 比 5% を超えたら v2 着手。下回ればコンセプト見直しに戻る（撤退容易性確保）。

v2 で追加:
- `route_learning_sessions` テーブル（修正版 SQL は章 5）
- JSONB の Zod スキーマ（`lib/route-learning-session-schema.ts`）
- AR 内チェックリスト・最大 3 問クイズ
- `children` テーブル正規化（`user_routes.child_id` text の置き換え）
- DPIA 拡充

### 3.3 v3 以降の方針

教育委員会監修対応、複数言語対応、ふりがな自動生成、TTS 読み上げ等。v2 後の利用データに基づき判断。

## 4. v1 MVP 仕様

### 4.1 機能要件

地図画面のARパネルに「親子で通学路確認」ボタンを追加する。対象は選択中の登録済み通学路のみ。選択ルートがない場合は通学路の選択または登録を促す。

通常 AR ビュー（現在地周辺の危険報告）は既存機能として残す。親子モードでは選択ルートの 100m 圏内にある危険ポイントだけを表示し、50m 以内に近づくと低学年向け注意文を強調する。

各危険ポイントで「確認した」「あとで見返す」を押下できる。状態は IndexedDB に保存し、ルート再選択や画面回転で消えない。

歩行終了時、`/route-quiz?routeId=xxx` への誘導ボタンを表示する（AR 内クイズは v2 で実装）。

子ども向け注意文の表示例:
- ここでは止まって、みぎ・ひだりを見よう
- この交差点は車から見えにくいよ
- 車が止まったことを見てからわたろう
- 歩道がせまいから、車道に近づきすぎないようにしよう
- 雨の日はすべりやすいから、ゆっくり歩こう

危険種別ごとの確認方針:
- 交通危険: 止まる、左右を見る、車が止まったか確認する
- 歩行者危険: 歩道の幅、車道との距離、自転車の通り道を見る
- 犯罪・不審者リスク: 人目のある場所、逃げ込める場所を確認する
- 災害・環境リスク: 雨の日、水たまり、暗い場所、避難しやすい方向を見る
- その他: 立ち止まる位置と周囲から見えやすいかを確認する

### 4.2 既存資産の活用方針

既存実装と重複させないため、以下を厳守する。

- `lib/ar-learning-tour.ts`（既存 127 行、`createARLearningContent` / `buildARLearningTourStops` / `summarizeARLearningTour`）は **不変**。保護者向け文言・ツアー stops 構築・完了時集計に引き続き利用。
- 子ども向け文言は **新規** `lib/ar-learning-tour-kids.ts` に分離する。関数: `createKidsHazardCue(report) -> { shortMessage, action, dangerKind }`。低学年向け語彙の独立辞書を持ち、既存 `BASE_CONTENT` を参照しない。
- ツアー stops のステータス（`pending` / `reviewed` / `saved`）は既存型を流用し、新ステータスを追加しない。仕様の「確認した」「あとで見返す」と 1:1 対応。
- `hooks/use-route-dangers.ts` の `bufferMeters=100` をデフォルトに使用（turf bufferベース）。
- 50m 強調判定は `lib/ar-utils.ts` の `calculateDistance`（haversine）を `useMemo` で利用する。turf.js を新たに追加しない（bundle size 節約）。
- `app/route-quiz/page.tsx` は v1 では誘導リンクのみ。AR 内クイズは v2。

### 4.3 ARView の型設計（discriminated union）

`mode` を string union ではなく discriminated union で表現し、`routeId / childId / sessionId / hazards` を mode ごとに型強制する。

```typescript
// lib/ar-view-types.ts (新規)
import type { DangerReport } from "@/lib/types"

export type ARViewMode =
  | { kind: "nearby"; reports: DangerReport[] }
  | {
      kind: "parent_child_route"
      routeId: string                 // user_routes.id
      routeName: string
      childId: string | null
      childName: string | null
      reports: DangerReport[]         // 選択ルート100m圏で既に filter 済み
      sessionId: string               // クライアント生成 uuid (idempotency)
    }

export interface ARViewProps {
  mode: ARViewMode
  onClose: () => void
}

// 使用側
function ARView({ mode, onClose }: ARViewProps) {
  if (mode.kind === "nearby") return <NearbyARView reports={mode.reports} onClose={onClose} />
  return <ParentChildARView {...mode} onClose={onClose} />
}
```

既存呼び出し（`map-container.tsx`）は `<ARView mode={{ kind: "nearby", reports: combinedReports }} onClose={...} />` への一括置換のみ。Boolean prop 爆発を防止。

親子モードでも `map-container.tsx` から渡すのは `DangerReport[]` とする。`ARHazardData` は現在地・方位・FOV 依存の派生データなので、`ARView` 内で `useARLocation` / `useAROrientation` の値を使って `calculateARHazardData` により算出する。

### 4.4 状態管理（IndexedDB セッション保存）

- AR セッション状態（reviewed/saved 件数、現在の stop、最終位置）を `useReducer` でローカルに保持する新フック `hooks/use-ar-learning-session.ts` を新設。
- 永続化は **IndexedDB**。実装時に `idb-keyval` を依存追加するか、`hooks/use-ar-learning-session.ts` 内に薄い IndexedDB ラッパーを持つ。キーは `routeId + sessionId`。アクションごとに段階保存（PATCH 連発でフレームごと書込はしない）。
- v2 で追加する Supabase 書込は **完了時 + 30 秒デバウンスの upsert** とし、`sessionId` を idempotency_key にする。同一ルートを 2 端末で同時開始しても両方残る。

### 4.5 GPS 精度ガード

- `watchPosition` の `accuracy > 50m` で「位置精度が低いため強調を抑制中」を表示し、50m 強調を一時停止。
- 移動速度 > 15km/h（自転車・車中検出）で接近通知を抑制。
- `PositionError.PERMISSION_DENIED` で手動「ここに着いた」ボタンへフォールバック。
- iOS Safari の `DeviceOrientationEvent.requestPermission` 拒否時は AR 表示を無効化し、「カメラ向きを使えないため、通常マップで確認してください」と誘導。
- 画面ロック・バックグラウンド復帰時はセンサーを停止/再開する（`visibilitychange` リスナ）。

### 4.6 計測イベント

既存の計測基盤がある場合は PostHog または GA4 に下記を送出する。未導入の場合は `lib/analytics.ts` の no-op ラッパーを先に作り、プロダクト判断で計測先を差し替えられる形にする。

- `ar_parent_mode_started` (route_id, child_id?)
- `ar_hazard_approached` (50m, hazard_id, danger_type)
- `ar_hazard_confirmed`（hazard_id）
- `ar_hazard_saved`（hazard_id）
- `ar_session_completed` (reviewed_count, saved_count, duration_s)
- `ar_quiz_link_clicked`（route_id） ※`/route-quiz` への遷移
- v2 追加: `ar_quiz_started` / `ar_quiz_completed` (score, total)

PII（`child_name`）は送出しない。Sentry breadcrumbs からも除外する。

### 4.7 アクセシビリティ要件

- タップ範囲は WCAG 2.5.5 準拠で **44x44px** 以上。Playwright の BoundingBox 検証を CI に組み込む。
- 屋外・直射日光下を想定し、コントラスト比は **AAA 7:1** を目標（半透明黒背景＋白文字）。
- 危険種別アイコンは色のみで分けず、形状＋ラベルを併記（色弱対応）。
- スクリーンリーダー: 注意文には `role="alert"` + `aria-live="assertive"`、ただし throttle して連発させない。
- 文字サイズ 18px 以上、行間 1.6、漢字は小2 までを目安（v3 で完全ふりがな対応）。
- `prefers-reduced-motion` で強調アニメーションを抑制。

## 5. v2 拡張仕様

### 5.1 データモデル（修正版 `route_learning_sessions`）

```sql
create table public.route_learning_sessions (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  route_id uuid not null references public.user_routes(id) on delete cascade,
  child_id text check (child_id is null or child_id ~ '^[A-Za-z0-9_-]{1,32}$'),
  child_name text check (child_name is null or char_length(child_name) <= 40),
  schema_version smallint not null default 1,
  started_at timestamptz not null default now(),
  completed_at timestamptz check (completed_at is null or completed_at >= started_at),
  reviewed_count integer not null default 0 check (reviewed_count >= 0),
  saved_count integer not null default 0 check (saved_count >= 0),
  quiz_score integer not null default 0 check (quiz_score >= 0),
  quiz_total integer not null default 0 check (quiz_total between 0 and 3),
  checklist jsonb not null default '[]'::jsonb check (jsonb_typeof(checklist) = 'array'),
  stop_results jsonb not null default '[]'::jsonb check (jsonb_typeof(stop_results) = 'array'),
  constraint quiz_score_le_total check (quiz_score <= quiz_total)
);

create index route_learning_sessions_user_route_started_idx
  on public.route_learning_sessions (user_id, route_id, started_at desc);

create index route_learning_sessions_route_id_idx
  on public.route_learning_sessions (route_id);
```

修正点（v1.0 仕様からの差分）:
- `child_id` に正規表現 CHECK（不正値混入防止）、`child_name` に長さ CHECK
- `completed_at >= started_at` の不変条件 CHECK
- `reviewed_count`, `saved_count`, `quiz_score`, `quiz_total` の範囲 CHECK
- `quiz_score <= quiz_total` の整合 CHECK（除算ゼロ・誤集計予防）
- JSONB 列に `jsonb_typeof = 'array'` CHECK
- `schema_version smallint` で将来の JSONB 変更に備える
- `route_id` は FK かつ cascade 対象のため単独インデックスを作る

### 5.2 RLS ポリシー（既存規約準拠 + IDOR 防止 WITH CHECK）

```sql
alter table public.route_learning_sessions enable row level security;

create policy "route_learning_sessions_select"
  on public.route_learning_sessions
  for select
  to authenticated
  using ((select auth.uid()) = user_id);

-- IDOR 防止: route_id の所有権を WITH CHECK で強制
create policy "route_learning_sessions_insert"
  on public.route_learning_sessions
  for insert
  to authenticated
  with check (
    (select auth.uid()) = user_id
    and exists (
      select 1 from public.user_routes r
      where r.id = route_id and r.user_id = (select auth.uid())
    )
  );

create policy "route_learning_sessions_update"
  on public.route_learning_sessions
  for update
  to authenticated
  using ((select auth.uid()) = user_id)
  with check ((select auth.uid()) = user_id);

create policy "route_learning_sessions_delete"
  on public.route_learning_sessions
  for delete
  to authenticated
  using ((select auth.uid()) = user_id);
```

修正点:
- `to public` → `to authenticated`（anon に空配列を返す無駄を排除）
- `auth.uid()` の直接評価を避け、既存マイグレーションでも使われている `(select auth.uid())` パターンで統一
- INSERT 時に `route_id` の所有権を WITH CHECK で強制（他人のルートに紐づける IDOR を阻止）

### 5.3 JSONB スキーマ（Zod 定義）

`lib/route-learning-session-schema.ts` を新設し、サーバ側書込前に必ず `parse` する。

```typescript
import { z } from "zod"

export const ChecklistItemSchema = z.object({
  id: z.string().min(1).max(64),
  label: z.string().min(1).max(120),
  checked: z.boolean(),
})

export const StopResultSchema = z.object({
  hazardId: z.string().uuid(),
  status: z.enum(["pending", "reviewed", "saved"]),
  approachedAt: z.string().datetime().optional(),
  resolvedAt: z.string().datetime().optional(),
})

export const SessionPayloadSchema = z.object({
  schemaVersion: z.literal(1),
  checklist: z.array(ChecklistItemSchema).max(20),
  stopResults: z.array(StopResultSchema).max(200),
})
```

不正な JSON が DB に到達する前に reject する。バージョン違いペイロードは `schemaVersion` ミスマッチで拒否し、サーバ側で migration 戦略を取る。

### 5.4 AR 内チェックリスト・最大 3 問クイズ

歩行終了後、AR 画面内で振り返りを表示する。

チェックリスト例:
- 危ない場所で止まる場所を確認した
- 車や自転車を見る方向を確認した
- 困ったときに行ける場所を確認した

クイズは選択ルート上の危険ポイントから自動生成する。`generateKidsQuiz(stops, { seed }): KidsQuiz` を純粋関数として実装し、選択肢シャッフルは seed 注入で決定的に。正答位置の偏り（chi^2 近似）を Vitest で検証。

クイズ例:
- 見通しが悪い交差点では、まず何をする？
  - 止まって左右を見る（正答）
  - 走ってわたる
  - スマホを見る
- 歩道がせまい道では、どこを歩く？
  - 車道からはなれて歩く（正答）
  - 車道の近くを歩く
  - 友だちと横に広がる

危険ポイントが 0 件の場合: 汎用 3 問にフォールバック。1〜2 件: 件数分のみ生成。

## 6. 非機能要件

### 6.1 性能
- AR 描画 16ms/frame 以内（危険ポイント 100/500/1000 件で Vitest bench 計測）
- 接近検知ループは setInterval 1Hz（`requestAnimationFrame` 内で全件 haversine しない）
- Supabase 書込み（v2）: 歩行 5 分間で **≤ 10 回**。500ms debounce + 完了時 upsert
- 初回ロード 3 秒以内（Android Go クラス低スペック端末）

### 6.2 可用性
- Supabase 障害時は IndexedDB に保持し、再送キューで自動リトライ
- トークン失効時も IndexedDB から復元
- オフライン時は手動「ここに着いた」モードへフォールバック

### 6.3 セキュリティ
- `child_name` は React 既定エスケープに依存。`dangerouslySetInnerHTML` 使用箇所は grep で全件確認
- Sentry / 計測イベントから PII（`child_name`、生 GPS 座標）を除外
- IDOR 防止: 別ユーザー `route_id` で INSERT すると 403 になることを統合テスト
- XSS テスト: `child_name` に `<script>`・絵文字・制御文字・256 字以上を投入

### 6.4 保守性
- 文言バージョニング: `schema_version smallint`
- 教育委員会監修対応の文言改訂フローは v3 で整備
- 子ども向け辞書は `lib/ar-learning-tour-kids.ts` の定数として一元管理

### 6.5 ブラウザ互換
- iOS Safari 14+, Chrome Android 90+
- WebXR は必須としない（既存 AR 実装互換）
- HTTPS 必須
- DeviceOrientation 権限プロンプト失敗時の代替動線を必ず用意

## 8. 成功条件

機能要件:
- 登録済み通学路を選ぶと「親子で通学路確認」モードを開始できる
- AR 上に選択ルート 100m 圏内の危険ポイントだけが表示される
- 危険ポイントに 50m 以内で近づくと、子ども向け短い注意が強調表示される
- 危険ポイントごとに「確認した」「あとで見返す」を記録できる（v1: IndexedDB / v2: Supabase）
- 歩行後にチェックリスト＋最大 3 問クイズを実施できる（v2）
- 通常 AR ビューの既存挙動は壊さない（回帰テスト必須）
- `pnpm typecheck` と関連 Vitest が通る
- 空テストファイル（`ar-view-learning-tour.test.tsx`, `ar-hazard-card-learning.test.tsx`）が解消されている

数値ゴール（v1 リリース後 30 日で測定）:
- `ar_parent_mode_started` の DAU 比 **5%** 超 → v2 着手判断
- 月 1 回以上モード起動するユーザーが登録親子の **30%** 以上
- 1 セッション平均確認ポイント数 **≥ 2**
- モード使用ユーザーの 7 日リテンション **≥ 非使用 +10pt**
- AR 内エラー率 **< 1%**

## 9. テスト観点

仕様書 v1.0 の 7 項目では境界値・GPS 異常系・並行性・IDOR を取りこぼすため、以下に拡充する。

### 9.1 境界値
1. 99m / 100m / 101m での包含・除外切替（float 誤差含む）
2. 49m / 50m / 51m での強調切替
3. クイズ生成元の危険ポイントが 0 / 1 / 2 / 3 / 4+ 件のとき最大 3 問保証
4. クイズ選択肢シャッフル（seed 固定で正答位置の偏りを chi^2 近似で検証）
5. 同一カテゴリ重複時の質問除外

### 9.2 異常系
6. GPS unavailable / `PositionError.PERMISSION_DENIED` でフォールバック
7. `accuracy > 50m` 時に強調抑制
8. iOS Safari `DeviceOrientationEvent.requestPermission` 拒否で AR 表示無効化＋誘導
9. ルート未選択でモード起動した際の導線（無限ローディングしない）
10. トークン失効時の保存リトライと IndexedDB 復元
11. 移動速度 > 15km/h 検出時の接近通知抑制

### 9.3 競合・データ整合性
12. 接近通知連発時の同一 stop 多重 reviewed 防止（debounce/idempotency）
13. ネットワーク断 → 復帰時のセッションマージ（`sessionId` を upsert キーに）
14. バックグラウンド復帰・画面ロック・タブ非アクティブ時のセンサー停止/再開

### 9.4 セキュリティ
15. 別ユーザーの `route_id` で INSERT → RLS で 403（v2）
16. `child_name` の XSS（`<script>`・絵文字・制御文字・256 字以上）
17. `stop_results` 巨大 JSON（>1MB）の拒否（v2）
18. Zod スキーマで不正ペイロード reject（v2）

### 9.5 回帰
19. `mode: { kind: "nearby" }` で既存 AR の表示要素・距離計算が変わらない
20. 空テストファイル 2 件（`ar-view-learning-tour.test.tsx`, `ar-hazard-card-learning.test.tsx`）が CI で 0 byte で残らない

### 9.6 アクセシビリティ
21. 「確認した」「あとで見返す」のタップ範囲が 44x44px 以上（Playwright BoundingBox）
22. 屋外想定で AAA 7:1 コントラスト
23. ふりがな対応・小2 漢字制限
24. `prefers-reduced-motion` で強調アニメ抑制

### 9.7 性能・負荷
25. 危険ポイント 100/500/1000 件で <16ms/frame（Vitest bench）
26. 歩行 5 分シナリオで Supabase 書込 ≤ 10 回（v2）

### 9.8 E2E（Playwright）
- `geolocation` を context override で擬似 GPS、`window.DeviceOrientationEvent` をスタブ
- ルート選択 → モード開始 → stop 近接 → 確認 → クイズ誘導クリック → 通常 AR が壊れていない
- ログアウト状態で `/map?mode=parent_child_route` 直リンク → `/login?next=...` リダイレクト

## 12. 前提

- 本機能はログインユーザー向け
- 対象範囲は選択中の登録済み通学路の **100m 圏内**
- 注意の強調表示は現在地から **50m 以内**
- 子ども向け文言は **低学年向け**（小1〜小3 想定、漢字は小2 まで）
- 既存 `/route-quiz` ページは v1 では別導線として残し、AR からはリンクで誘導。v2 で統合可否を再判断
- CI に `find tests -name '*.test.*' -size 0 -exec false {} +` ステップを追加し、空テストファイルが残らない
- `package.json` の `test:components` script にワーカー待機時間を固定する場合は、Windows互換のため `cross-env VITEST_MAX_WORKER_THREADS_WAIT=120000 ...` 形式にする
- 既存 RLS 規約は `(select auth.uid()) = user_id` パターン。新規ポリシーもこれに準拠
- `lib/ar-learning-tour.ts`（既存 127 行）と `app/route-quiz/page.tsx`（既存）は本機能で**変更しない**
- 法務確認（規約・PP・保護者同意・AR 起因事故警告 UI）は v1 リリース必須
