# Parent Route Simulation Growth Design

## Summary

この文書は、`小学生の保護者` を主軸にした PathGuardian の再設計案を定義する。
前提は変えない。ホームは引き続きニュース型で運用し、コア価値だけを強く差し込む。
今回の設計テーマは `うちの子に引き寄せた危険の実感化` であり、画像投稿・生成・共有を「家族の安全確認行動」に変換することが目的である。

## Persona

- 38歳、共働き、小学生1年生の母親
- 朝は時間がなく、長文や複雑な操作は避けたい
- 危険箇所の存在そのものより `明日の通学で子どもにどう伝えるか` を知りたい
- SNSで発信したいわけではないが、家族や保護者グループには役立つ情報をすぐ共有したい
- 投稿の動機は承認欲求ではなく `子どもを守りたい` と `他の親にも知らせたい`

## Product Goal

- 初回体験で `これはすごい、うちの子に使える` と思わせる
- 毎朝または見守り時に自然に開く理由を作る
- 投稿後に `共有したい` と感じる完成度を持たせる
- 既存のニュース型ホーム、危険報告、通学路管理、画像生成、通知、レポート生成を活かす

## Non-Goals

- ホーム全体をダッシュボード画面に置き換えない
- ランキングやゲーム感を前面に出さない
- 不特定多数向け SNS 拡散を主戦場にしない
- 大規模な新規基盤を前提にしない

## Core Idea

`うちの子通学シミュレーション`

危険投稿を単なる写真記録で終わらせず、投稿または閲覧時に以下の3点へ再構成する。

1. 子ども目線の生成画像
2. 5秒で理解できる危険要約
3. 明日の登校で使える回避アクション

これにより、保護者は `危険を見た` ではなく `家族で行動を変えられる` 状態に入る。

## User Story

### 朝

朝7:10、保護者はニュースを見にホームを開く。
ホーム最上段にだけ `今日の通学3分チェック` ユニットがあり、登録済み通学路に紐づく最新危険を3件以内で提示する。
表示項目は `今日の注意地点数` `条件別注意（雨、薄暮）` `直近の家族共有カード`。
ここで1件をタップすると、危険投稿の元写真に加え、子ども目線シミュレーションと一言アドバイスが見える。

### 夕方

夕方18:20、迎えや見守りの途中で見通しの悪い地点を発見する。
モバイル下部の `報告` CTA から撮影し、そのまま `今ここを報告` フローへ入る。
写真アップロード後、既存の生成基盤を使って `子ども目線` `危険要約` `家族共有カード用プレビュー` を返す。
ユーザーは `投稿する` 前に、他人向けではなく家族向けの意味づけを確認できる。

### 夜

投稿完了後は終わりではない。
完了画面に `家族に共有する` `通学路スコアの変化` `明日の注意アクション` を表示する。
共有されたカードは LINE 向けに最適化されており、夫婦や祖父母、保護者グループで再利用される。

## Current UX Problems

### 1. 投稿が「記録」で止まっている

今の体験は写真投稿、画像生成、一覧閲覧まではあるが、`わが子に関係ある危険` まで翻訳されない。
そのため初回での驚きが弱く、投稿の意味も伝わりにくい。

### 2. 投稿後の報酬が弱い

投稿完了後に次の行動が少ない。
今の構造では `保存された` 以上の感情変化が起きづらい。

### 3. 共有物が会話の起点になりにくい

共有導線は存在するが、保護者がそのまま送れる `1枚で伝わる状態` になっていない。
家族や保護者グループへの転送に耐える粒度が必要。

### 4. 毎日開く理由が弱い

ニュース型ホームは回遊しやすいが、保護者本人に直結する `今日の理由` が不足している。

## Information Architecture

### Home

ホームはニュース型のまま維持する。
ただし最上段にだけ `わが子の通学ダッシュボード` を固定挿入する。

推奨順序:

1. 今日の通学3分チェック
2. ヒーローカルーセル
3. 通学路ニュース
4. 危険マップ誘導
5. 既存の特集 / ストア / ヒヤリハット

### Dashboard Unit

ユニット内の情報は多くしない。
上限は3情報まで。

- 今日の注意地点
- 条件付き注意
- 直近共有カードへの再アクセス

### Report Flow

`撮る -> 生成を見る -> 投稿する -> 家族に共有する`

これをひと続きの一本線にする。
投稿完了後のアクションを別画面へ逃がさない。

## UX Design

### Screen 1: ホーム上部 `今日の通学3分チェック`

目的:
ニュース型ホームを壊さず、パーソナル価値を最初に見せる。

構成:

- 見出し: `今日の通学3分チェック`
- サブコピー: `登録した通学路の注意点を30秒で確認`
- 1枚目カード: `今日の注意地点 2件`
- 2枚目カード: `雨の日に見えにくい地点 1件`
- 3枚目カード: `昨夜家族に共有したカードを開く`
- CTA: `通学路を見る`

挙動:

- 未ログイン時は表示しない
- 通学路未登録時は `通学路を登録すると、わが子向けの注意が届きます` に差し替える
- 子どもが複数いる場合は既存の child route 概念を使って切り替える

### Screen 2: 危険地点詳細

目的:
危険の理解コストを下げる。

必須表示:

- 元写真
- 子ども目線シミュレーション
- 危険要約1文
- 回避アクション1文
- 発生時間 / 投稿日時 / 危険タイプ
- `家族に共有` CTA

禁止事項:

- 初期表示で長文説明を見せない
- コメントやリアクションを主役にしない

### Screen 3: 投稿フロー

目的:
保護者が 60 秒以内に投稿を完了できるようにする。

理想フロー:

1. 場所を選ぶ
2. 写真を撮る / 選ぶ
3. 危険タイプを選ぶ
4. 子ども目線シミュレーションを確認する
5. 投稿する

入力ルール:

- タイトルは自動仮入力を許容
- 説明は任意でも成立させる
- 生成結果の再試行はワンタップ
- 投稿前プレビューで `家族にどう伝わるか` を見せる

### Screen 4: 投稿完了

目的:
投稿直後の感情を最大化し、共有まで連れていく。

表示項目:

- 見出し: `子ども目線の注意カードができました`
- プレビュー: 共有カードの実物
- 通学路スコア: `72 -> 68`
- 明日の注意アクション: `交差点の5m手前で必ず止まる`
- CTA 1: `家族に共有`
- CTA 2: `通学路で確認`

削除するもの:

- `守れた家庭数` の表示は入れない

### Screen 5: 共有カード

目的:
LINE 転送率を上げる。

1枚カードに含めるもの:

- 地点名
- 元写真または生成画像の代表1枚
- `小学生の目線では車が突然見える`
- `右側の白線沿いに歩く` のような回避行動
- 小さな地図サムネ
- 更新時刻

トーン:

- 不安を煽りすぎない
- 指示が明快
- 文字は少ない

## Emotional Curve

### Before

- 危険写真を見ても、わが子にどこまで関係あるか分からない
- 投稿しても意味があるのか分からない
- 必要な時しか開かない

### After

- 明日の通学で何を言えばいいか分かる
- 投稿した内容をそのまま家族に渡せる
- 朝のチェックと見守り時に自然に開く

## Growth Mechanics

### Trigger

- 朝の登校前
- 学童帰りの見守り時
- 雨予報や薄暮
- 保護者グループ内での共有受信

### Action

- ホーム上部の通学ダッシュボードを見る
- 危険地点を閲覧する
- 現地で写真付き投稿をする
- 共有カードを送る

### Reward

- 自分の家庭に必要な注意点がすぐ分かる
- 投稿が家族連携に直結する
- 通学路スコアの変化で更新を実感できる

### Retention

- 今日の通学3分チェック
- 条件別注意
- 直近共有カードの再閲覧
- 週次レポート

## KPI

- 初回登録後 `通学路登録完了率` 70%
- 初回セッション `ダッシュボード閲覧率` 65%
- 初回セッション `シミュレーション閲覧率` 60%
- 月次 `危険投稿実行率` 28%
- 投稿完了後 `家族共有実行率` 35%
- `D7 継続率` 35%
- `D30 継続率` 18%
- ホーム来訪ユーザーの `週3回以上閲覧率` 40%
- 共有カード受信後 `再訪率` 20%以上

## Phase Plan

### Phase 1

- ホーム上部の通学ダッシュボード
- 投稿完了画面の再設計
- 家族共有カード

### Phase 2

- 条件別注意
- 週次レポート
- 子ども別切り替えの改善

### Phase 3

- PTA / 学校提出用エクスポート強化
- 共有カードのABテスト

## Implementation Constraints

- ホーム全体の構造は変えず、`landing` ページ内への差し込みで実装する
- 既存の danger report 画像生成フローを拡張し、別AI基盤は追加しない
- 共有は Web Share API とクリップボードを主軸にし、ネイティブ依存は避ける
- KPI はまず既存データ構造と軽量イベントログで取る

## Recommended Build Order

1. ホーム最上段の通学ダッシュボード
2. 投稿フロー内のシミュレーションプレビュー改善
3. 投稿完了画面の再設計
4. 家族共有カード
5. 条件別注意
6. 週次レポート
