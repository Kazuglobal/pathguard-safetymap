### 3. 対象ユーザー

- 小学生: 漢字が読めない、長文を読まない、44px未満を押しにくい、次の行動が分からないと離脱する

すべての意思決定で「初見ユーザーが説明なしに目的を完了できるか」を優先する。

### 4. 実装順序

#### Phase 0: ベースラインと変更計画

1. `git status --short` を確認し、既存変更を保護する。
2. 監査対象ルートと関連コンポーネントを特定する。
3. 既存テストを実行し、開始時点の失敗を記録する。
4. 変更対象、再利用する既存コンポーネント、追加テストを短い計画にする。
5. `tankenTokens`、既存の `PaperPanel`、`TankenButton`、マスコット、ナビゲーションを優先して再利用する。

対象:

要件:

#### Phase 2: Critical — 危険報告を完遂可能にする

正本画像:

- `mockups/01-report-primary-cta.png`
- `mockups/compare-01-report.png`

関連候補:

- `app/report/page.tsx`
- `components/danger-report/danger-report-form.tsx`
- `components/map/map-container.tsx`
- `components/map/map-floating-controls.tsx`
- `components/danger-report/wizard/**`

要件:

1. `/report` 最上部へ主CTA「危険を報告する」を追加する。
2. CTA直下に `1 場所 → 2 写真・内容 → 3 確認` を表示する。
3. 二次CTA「現在地からはじめる」を置く。
4. `/report` と `/map` の報告ボタンは、同じウィザード起動処理を使う。
5. Mapboxの完全読込をウィザード開始条件にしない。地図が遅い場合も場所検索・現在地・後から調整で開始できる。
6. GPS拒否時は「住所や地図から選ぶ」を提示する。
7. 写真拒否／未添付でも、写真が任意なら先へ進める。必須なら理由と代替手段を示す。
8. 各ステップで戻る、入力保持、送信中、成功、失敗、再試行を実装する。
9. 最終送信前に、写真の外部AI送信について現在の説明を維持する。
10. 成功後は報告ID、審査状態、地図／履歴への戻り先を表示する。

正確に使う主要文言:

- `危険を報告する`
- `現在地からはじめる`
- `1 場所`
- `2 写真・内容`
- `3 確認`

テスト:

- GPS許可、拒否、タイムアウト
- 写真あり／なし、非画像、サイズ超過
- 必須エラー、戻る、再送
- `[UXテスト]` 投稿の成功

#### Phase 4: High — 地図の段階的ローディング

正本画像:

- `mockups/03-map-progressive-loading.png`
- `mockups/compare-03-map-loading.png`

要件:

1. スピナーだけの状態を廃止する。
2. ベース地図、危険マーカー、ユーザー情報の進捗を段階表示する。

### 5. 共通UI要件

- 色、角丸、影、フォーカスは `tankenTokens` を使う
- Zen Maru Gothic系の既存設定を維持
- 主タップ領域は44×44px以上。子ども向け主CTAは高さ52〜58pxを推奨
- アイコンだけの操作には日本語の `aria-label` を付ける
- エラーは色だけで示さず、アイコン＋短い文言を使う
- ローディング、成功、失敗は `aria-live` で通知する
- キーボードフォーカスは `tankenTokens.cls.focus` を使い、常に視認可能にする
- `prefers-reduced-motion` を尊重する
- モバイルで横スクロール、地図の極端な縮小、固定フッターとの重なりを起こさない
- 長文ページは見出し構造とランドマークを正しくする
- 子ども向け文言は、ひらがな優先、1文を短く、漢字には既存 `RubyText` を利用する

### 6. パフォーマンス／安定性

監査では長時間操作中に開発サーバーのメモリ再起動と `ECONNRESET` が2回発生した。ただし監査環境はNode.js 24で、リポジトリの指定はNode.js 20だった。

1. まずNode.js 20で再現確認し、環境差と製品側問題を分ける。
2. Mapbox、Cesium、MediaPipe、Three.jsの重複初期化と破棄漏れを調べる。
3. 画面遷移後にWebGLコンテキスト、イベントリスナー、Object URL、タイマー、AbortControllerが解放されることを確認する。
4. 大規模データや画像をコンポーネント状態へ保持し続けない。
5. パフォーマンス修正をUX変更と混ぜず、計測結果を残す。

### 7. テスト方針

変更前に失敗するテストを追加し、変更後に通す。

最低限:

- ユニット／コンポーネントテスト
  - adminロール判定
  - dashboardリダイレクト防止
  - ウィザード状態遷移と入力保持
  - 登録フォームのエラー写像
  - 地図ローディング状態
  - ハンターのタイムアウトとキャンセル
- E2E
  - 390×844と1280×900
  - LP→登録→ログイン
  - 地図フィルター→マーカー詳細
  - `/report` CTA→3ステップ→成功
  - route quizで2地点→開始
  - ハンターで写真選択→処理→再試行
  - dashboard／mypage
  - admin 3ルート
  - ニュース／マガジン一覧→詳細→戻る
- アクセシビリティ
  - Tab順序
  - フォーカス可視性
  - フォームラベル、エラー関連付け
  - 44pxターゲット
  - reduced motion

外部AI、実メール、実通知をE2Eから直接呼ばない。ローカルテストではモック／テストダブルを使う。Supabase統合テストが必要なら既知テストユーザーだけを使い、作成データへ `[UXテスト]` を付ける。

### 8. ビジュアル検証

1. 390×844と1280×900で対象画面を撮影する。
2. 承認済み比較画像と同じ状態・同じ幅で並べる。
3. 次を目視確認する。
   - 情報の順序
   - CTA文言と優先順位
   - クリーム／緑／オレンジ／黄
   - Zen Maru Gothic系
   - 角丸、余白、カード影
   - 固定ナビとの重なり
   - 小さい文字や切れた日本語がないこと
4. 完全一致より、実データとアクセシビリティを優先する。ただし承認済みの情報階層を変更しない。

### 9. 完了条件

- Critical 4件がすべて完遂可能
- 承認済み6画面が実装され、両幅で破綻しない
- トップ10の残りも対応または明確な未対応理由がある
