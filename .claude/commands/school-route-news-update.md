# 通学路の安全ニュース 日次更新（デイリーハビット設計 v3.1）

全国で起きた通学路関連のニュースを取得・検証・記事化し、保護者が「毎朝1分で今日の見守りに必要なことが全部わかる」状態を作るワークフローを実行します。

---

## 編集ミッション（すべての判断の最上位基準）

通学路の安全ニュースは「不審者速報の垂れ流し」ではなく、**保護者が毎朝開いて確認し、安心して1日を始めるためのデイリーチェック習慣**である。SAFE MAGAZINE（週次・解説）と異なり、こちらは**速報・日次・客観的**。ただし次の4原則は共通の憲法として厳守する。

### デイリーハビット4原則

1. **トリガーの固定**: 更新は毎朝7:00に完了させ、7:30のダイジェスト通知に間に合わせる。配信時刻のブレは習慣の錨を壊す
2. **完了感の保証**: 1日の新規公開は最大5本。読み切れる有限のフィードが「今日のチェックは以上です」を成立させる。量より「今日の要点」
3. **自分ごと化**: すべての記事に正規化された地域情報を付与する（後述の地域整合性ルール）。地域絞込が壊れる記事は公開しない
4. **恐怖で終わらせない**: 危険を伝える記事は必ず「そなえの一言」で締める。恐怖だけ残して閉じさせる記事は公開禁止

---

## 必須ルール（厳守）

### ルール1: 事実のみを記載する
- 実際に検索・取得して確認できた情報のみを使用する。架空・推測・要約の捏造は禁止
- 出典URLが実在し、内容が確認できるもののみを記事化する
- 複数ソースまたは公式発表（警察・自治体・防犯メールアーカイブ）で裏付けが取れない事案は記事化しない
- 統計・件数には「収集範囲・期間」を必ず添える

### ルール2: 個人特定情報の禁止
- 被害児童の学年・氏名・学校名・詳細な自宅位置は記載しない
- 発生場所は「市区町村＋町名」まで。番地・目印になる個人宅情報は書かない

### ルール3: そなえの一言（不安と安心のバランス・必須）
- `category` が `accident` / `suspicious` の記事は、`actionAdvice` フィールドに「今日からできる具体行動1つ」を必ず設定する
- フォーマット: **30〜60字・1行・具体的な動作**（「気をつけましょう」禁止）
  - 良い例: 「防犯ブザーが鳴るか・電池が残っているかを今夜いっしょに確認する」
  - 悪い例: 「不審者に注意しましょう」
- 語彙はチーム設定JSONの `action_phrase_library` を参照してよいが、事案の手口に対応した行動を優先する（声かけ事案→声かけ対応の行動）
- **不審者の身体的特徴・服装・手口などの防犯上必要な情報は削らない**。特徴の完全な記載と、そなえの一言の併記は両立させる
- `keyPoints` は「①最重要ファクト ②そなえの一言」の順で書く（一覧UIは先頭2件を表示するため）

### ルール4: 地域整合性ルール（パーソナライズの生命線）
- `location.prefecture` は47都道府県の**正式名称**（「東京都」「大阪府」「北海道」「福岡県」等）または「全国」のみ。略称・ひらがな・欠落は禁止
- `location.city` は市区町村の正式名称（「北九州市」「函館市」）。政令市の区は `area` に入れる
- この正規化が壊れると地域フィルター・地域別Push通知・ダイジェスト集計のすべてが壊れる。公開前監査（Step 6）の必須チェック項目

### ルール5: 重複記事を作成しない
- 記事作成前に `lib/school-route-news.ts` の `NEWS_ITEMS` を確認し、同一事案・同一発表の重複記事化を禁止する
- 例外＝続報: 新しい事実（続報発表・検挙・対策実施）がある場合のみ、前回記事に言及した上で記事化してよい

## データ形式

`lib/school-route-news.ts` の `NEWS_ITEMS` 配列に追加する。既存フィールドは従来通り。**v3で追加されたoptionalフィールド**に注意:

```typescript
{
  id: "news-YYYY-MM-DD-NNN",
  slug: "prefecture-type-YYYYMMDD",
  title: "【都道府県名】タイトル（50文字以内）",
  excerpt: "150文字以内の要約",
  content: "Markdown形式の本文（末尾は必ず「そなえの一言」または前向きな情報で締める）",
  category: "accident" | "suspicious" | "infrastructure" | "policy" | "community",
  categoryLabel: "カテゴリー名",
  categoryColor: "#カラーコード",
  categoryIcon: "アイコン名",
  publishedDate: "ISO8601形式（JST）",
  location: {
    prefecture: "47都道府県の正式名称 or 全国",   // ルール4厳守
    city: "市区町村の正式名称",
    area: "町名・区名"
  },
  tags: ["タグ1", "タグ2"],
  sources: ["ソース1", "ソース2"],
  keyPoints: ["最重要ファクト", "そなえの一言"],
  thumbnailUrl: "/images/school-route-news/thumbnails/{slug}.png",
  isBreaking: true/false,
  verifiedAt: "ISO8601形式",

  // ---- v3 追加（すべてoptional・後方互換）----
  actionAdvice: "そなえの一言（accident/suspiciousでは必須）",
  newsType: "daily" | "weekly_trend"   // 省略時 daily。週次ロールアップのみ weekly_trend
}
```

**ランディングページ用の更新作業は不要**（`lib/landing-news-preview.ts` は `NEWS_ITEMS` からの自動導出。手動配列は廃止済み）。

## 日次ワークフロー

$ARGUMENTS にカテゴリ指定があればそのカテゴリを優先。未指定なら全カテゴリを対象とする。

### Step 0: 既存データとLayer 2クラスタの確認（必須）
1. `NEWS_ITEMS` の直近記事を確認し、重複トピックを除外する
2. **既知事案リストの作成**: `NEWS_ITEMS` の直近14日分から `[都道府県, 発生日, カテゴリ, 概要]` を書き出す。Step 1 の検索結果はこのリストと**先出しで**突合する（記事を書いてから重複に気づく後追い照合を禁止）
3. **要追跡リストの引き継ぎ**: 前回の作業レポート末尾に「要追跡」（容体続報待ち・検挙発表待ち等）があれば、まずその続報から検索する
4. 90日より古い記事を配列から削除する（保持期間ポリシー）
5. Supabase `local_safety_alerts` を確認し、**同一都道府県・同一カテゴリで直近7日間に3件以上**のクラスタがあれば、「地域傾向記事」の候補としてStep 3へ回す（個別速報の再掲ではなく、傾向＋そなえとして記事化する）

### Step 1: ニュース検索（news-fetcher）

**交通事故:** 「通学路 事故 速報 2026」「スクールゾーン 交通事故」「登下校 児童 事故」「横断歩道 小学生」
**不審者情報:** 「通学路 不審者 2026」「児童 声かけ事案」「登下校 つきまとい」
**インフラ整備:** 「通学路 ガードレール 設置」「スクールゾーン 整備」「歩道 拡幅 工事」
**施策:** 「通学路 安全対策 自治体」「見守り活動 強化」
**地域活動:** 「見守りボランティア」「スクールガード 活動」

**バランス指針**: 1日の公開分に `suspicious` / `accident` しか無い状態を避ける。`infrastructure` / `policy` / `community` の前向きなニュースを最低1本探す努力をする（見つからない日は無理に作らない——ルール1優先）。

#### Step 1-a: 多角的クエリローテーション（漏れ防止・特に accident）

`accident` は Layer 2（Supabase 2時間毎cron）の自動収集対象外で手動検索が唯一の入口のため、1〜2語の検索で済ませることを禁止し、以下の軸を掛け合わせて**最低8パターン以上**検索する:

- **事案語**: 通学路 / 登校中 / 下校中 / 集団登校 / スクールゾーン / 横断歩道 / 児童
- **結果語**: 事故 / はねられる / ひき逃げ / 死亡 / 重体 / 重傷 / 巻き込まれる
- **時間軸**: 本日 / 昨日 / 今週 / 「YYYY年M月」（実行月）
- **地域軸**: 直近30日で記事が0件の都道府県を優先的に地域名指定で検索する（全国一括の曖昧検索では地方紙のローカル報道が拾えない）

**情報源の優先順位**:
1. 都道府県警の交通事故発生状況・報道発表（公式）
2. 自治体・教育委員会の公式発表、防犯/安全メールのアーカイブ
3. 地方紙・地域ニュース（通学路事故は全国紙より地元紙が詳報する）
4. 全国紙・NHK地域面
5. Yahoo!ニュース地域トピックス

#### Step 1-b: 差分照合と検索ログ（「探していない」と「無かった」の区別）

1. 検索結果を Step 0 の既知事案リストと突合し、重複を除外する
2. 検索したが新規候補が0件だった地域・クエリは「**検索したが該当なし**」として検索ログに明記する（次回実行時に、漏れなのか本当に無かったのかを区別できるようにする）
3. 情報源が1つしか見つからない事案は「未確認」として保留し、記事化しない（ルール1）。要追跡リストに載せて翌日再確認する

### Step 2: ファクトチェック（fact-checker）
1. ソースの信頼性を確認（警察・自治体の公式発表、防犯メールアーカイブ、大手メディアを優先）
2. 複数ソースでクロスチェックし、日時・場所・詳細の正確性を確認
3. 誤報・デマ・未確認情報を除外
4. 個人特定情報（ルール2）が含まれていないか確認

#### Step 2-a: 時点情報の再検証（続報・容体・法的ステータス）

> 背景: 2026-07-14 の浜松事故記事で「意識回復」を未確認のまま記載し、後日削除する訂正が発生した（コミット a2a6c8f59）。同じ誤りを構造的に防ぐ。

1. **被害者の容体**（意識の有無・退院・後遺症等）は、情報源が明記し、かつその発表日時が**現時点で最新**であることを確認できた場合のみ記載する
2. 発表から時間が経っている容体情報は「〇月〇日時点で」と時点を明記する。最新かどうか確認できない場合は**記載しない**（安全側に倒す）
3. 検挙・送検・処分結果などの法的ステータスも同様に、最新発表のみを採用する
4. 続報記事では、前回記事に書いた容体・ステータスが古くなっていないかも再検証する

### Step 3: 記事作成（news-writer）
- 上記データ形式に従い執筆。`accident` / `suspicious` は `actionAdvice` 必須
- タイトル公式: `【都道府県＋市区町村】＋ 何が起きたか ＋（特徴があれば要点1つ）`
- 本文構成: 事案の概要 → 特徴・手口（防犯上必要な情報は省略しない）→ 発生時間帯・場所の文脈 → **そなえの一言**（見出し付きで独立させる）→ 出典
- トーン: 客観的・冷静。保護者を責めない。煽らない

### Step 4: SEO最適化（seo-optimizer）
- メタタイトル（60字以内）・メタディスクリプション（160字以内）
- 地域名＋日付のキーワード最適化、JSON-LD構造化データ

### Step 5: 画像生成（image-generator）
- 事故現場・不審者本人を直接描かない。抽象的・注意喚起的な表現
- 日本の通学シーン要素（黄色い帽子・ランドセル・横断歩道・見守り）
- カテゴリカラーに合わせたトーン（accident=赤系警告 / suspicious=オレンジ警戒 / infrastructure=青の安心 / community=緑の温かみ）
- 出力先: `public/images/school-route-news/thumbnails/{slug}.png`
- **サムネイルの使い回し禁止**。必ず当該記事のslugで新規生成する

### Step 6: 公開前監査（regional-coverage-auditor・必須ゲート）
以下をすべて確認し、1つでも不合格なら該当Stepへ差し戻す:
- [ ] `location.prefecture` が47都道府県の正式名称 or「全国」
- [ ] `accident` / `suspicious` 記事に `actionAdvice` がある
- [ ] 本文が恐怖・不安の記述で終わっていない
- [ ] 個人特定情報が含まれていない
- [ ] `thumbnailUrl` のファイルが実在し、他記事の画像を参照していない
- [ ] `NEWS_ITEMS` 内でslug・idが一意
- [ ] 1日の新規公開が5本以内

### Step 7: 検証ゲート（緑になるまで完了と言わない）
```bash
npx tsc --noEmit
npx vitest run tests/unit/lib/school-route-news-release-readiness.test.ts tests/unit/lib/school-route-news-format.test.ts
```
その後、`/school-route-news` と `/landing` をブラウザで確認（ランディングは自動導出なので、一覧に新記事が出ていれば導出が機能している）。

### Step 8: 作業レポート（証拠つき）
- 公開記事一覧（カテゴリ・地域・actionAdviceの内容）
- **実行した検索クエリの一覧**（何を検索したかの証拠。Step 1-a の8パターン以上を満たしているか自己点検）
- **「検索したが該当なし」だった地域・クエリのリスト**（Step 1-b の検索ログ）
- 除外した候補と除外理由（ファクトチェック落ち・重複・単一ソース未確認等）
- Layer 2クラスタの検知状況（傾向記事化した/しなかったの判断込み）
- **要追跡リスト**（容体続報待ち・検挙発表待ち・単一ソース保留の事案。次回実行のStep 0で必ず引き継ぐ）

## 週次ワークフロー（毎週月曜のみ・日次に追加で実行）

### Step W1: 週次集計（trend-analyst）
1. Supabase `local_safety_alerts` の直近7日分を集計: カテゴリ別件数（前週比）、都道府県分布、`occurred_at` の時間帯分布
2. `NEWS_ITEMS` の直近7日分のカテゴリ内訳を集計

### Step W2: 傾向記事の執筆（trend-analyst → fact-checker検算）
- 毎週同じ骨格で書く（定点観測の型が再訪の理由になる）:
  1. 今週の総件数とカテゴリ内訳（前週比）
  2. 目立った傾向を**1つだけ**（例: 「声かけ事案の6割が15〜17時台」）
  3. 傾向に対応した「今週のそなえ」1つ
  4. アプリの地域フィルターへの導線
- `newsType: "weekly_trend"`、`tags` に `週次傾向`、`category` はその週の支配的カテゴリ
- **過剰一般化の禁止**: 母数と収集範囲の限界を必ず明記（「本アプリが収集した公開情報の範囲では」）。小さいnで「全国的に急増」と書かない
- fact-checkerが集計数値を再計算して検算する

### Step W3: 画像生成・公開前監査・検証ゲート
日次のStep 5〜7と同一。

## Push通知の運用（実装済みCronとの関係）

- **Layer 2個別速報**（既存）: `/api/cron/local-safety-alerts`（2時間毎）が `suspicious` / `voice_call` を配信。JST 22:00〜翌7:30発生分は個別配信せず翌朝ダイジェストへ回す。同一都道府県で1サイクル3件以上は1通に束ねる
- **朝のダイジェスト**（新設）: `/api/cron/daily-news-digest`（`30 22 * * *` UTC = JST 7:30）が「今日は全国でX件」を配信。プリファレンスキーは `daily_digest`（`local_alerts` とは独立）。送信時に夜間の未通知アラートの `push_notified_at` を埋め、二重通知を構造的に防ぐ
- このコマンド（コンテンツ更新）は通知を直接送らない。**7:00までに `NEWS_ITEMS` 更新を完了させること**が通知品質への責務

## 効果測定の指標

| 区分 | 指標 | 見るもの |
|------|------|---------|
| 定量 | 7:30通知からの開封率 | トリガー設計の機能 |
| 定量 | 終端カード到達率 | 完了感の成立（フィードが読み切れる量か） |
| 定量 | 7日以内の再訪率・連続開封日数 | 習慣化の進み具合 |
| 定量 | 地域フィルター利用率 | 自分ごと化の成立 |
| 定量 | 通知オプトアウト率（daily_digest / local_alerts別） | 通知疲れの早期検知 |
| 定性 | 「確認して安心できたか」フィードバック | 恐怖に頼らない設計の検証 |
