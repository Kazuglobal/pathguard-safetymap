# UX総点検→改善画像案→実装指示書ワークフロー

## なぜこの順序か

UI/UXの「使いづらさ」は実装者の頭の中では再現できない。実際にユーザーとして操作した証拠(スクリーンショット)に基づかない指摘は的外れになりやすく、いきなりコードを直すと「直したが使いやすくなっていない」手戻りが起きる。そのため **操作の証拠 → 整理 → デザイン案の合意 → 実装** の順を厳守し、各フェーズにゲートを置く。

**最重要ゲート: Phase 3 の改善案画像にユーザーのOKが出るまで、アプリのコードは1行も変更しない。**

## フェーズ概要

| Phase | 内容 | 成果物 | ゲート |
|---|---|---|---|
| 0 | 起動と準備 | dev サーバー稼働・ログイン成功 | 全対象ページが開けること |
| 1 | ペルソナ・ウォークスルー | 操作スクショ一式 | 全対象ルート×2画面幅を実操作済み |
| 2 | 監査整理 | `docs/review/ux-audit-<日付>.md` | 重大度つき指摘+優先順位トップ10 |
| 3 | 改善案の可視化 | 現状vs改善の画像対比(5〜8画面) | **ユーザーのOK(ここで必ず停止)** |
| 4 | 実装指示書 | 実装用プロンプト/タスク分割 | ユーザー承認後のみ着手 |

依頼が「監査だけ」「改善案だけ」のように部分的な場合も、前フェーズの成果物(直近の監査レポート等)が存在するかを先に確認し、なければそのフェーズから始める。

## Phase 0: 起動と準備

- 起動: `pnpm dev`(predev が Cesium/MediaPipe アセットをコピーしてから `next dev --webpack` を起動)。URL は http://localhost:3000
- **`/` は製品紹介LP(`/lp`)へリダイレクトされる仕様**。不具合として報告しない
- ログイン: `/login` からテストユーザーで。認証情報は `scripts/create-test-users.ts` に定義済み。Supabase 側に未作成なら `pnpm create-test-users`(.env.local 必須。**ファイル内容は読まない**)
- **ハイドレーションが遅い**: ページ遷移後は最低6秒待ってから操作・スクショする。地図ページは Mapbox タイル読み込みでさらに数秒かかる
- Playwright を使う場合は `channel: 'chrome'` を指定する(過去の実測でヘッドレス既定ブラウザだと描画が不安定)

## Phase 1: ペルソナ・ウォークスルー

次の2ペルソナに**なりきって**、タスクベースで操作する。単なる表示確認で終わらせない。「初見ユーザーがこの目的を達成できるか」を毎回問う。

- **小学生の子ども**: 漢字が読めない/長文を読まない/タップ領域が小さいと押せない/次に何をすればいいか迷うと離脱する
- **保護者(30-40代、ITリテラシー普通)**: 子どもの安全情報を素早く確認したい/手順が多いと挫折する

### 対象ルートと検証タスク

ルート一覧は変わりうるので、着手時に `app/**/page.tsx` を列挙して下表と突き合わせ、増えた画面は適切な優先度で追加する。

| 優先 | ルート | 検証タスク |
|---|---|---|
| 高 | `/lp` → `/register` → `/login` | 初見で登録〜ログイン完了までの導線 |
| 高 | `/map` | 危険箇所を見る/フィルタ/マーカー詳細を開く(メイン画面) |
| 高 | `/report` | 危険箇所を報告(3ステップウィザード)。写真添付・現在地取得含む |
| 高 | `/safety-quest` → `/safety-quest/hunter` | きけんハンター。子ども目線で最重要 |
| 高 | `/dashboard` `/mypage` | 自分の状況(ポイント・報告履歴)が把握できるか |
| 中 | `/routes` `/route-quiz` | 通学路の登録・クイズ |
| 中 | `/school-route-news` `/safe-magazine` | 一覧→詳細の回遊 |
| 中 | `/missions` `/badges` `/leaderboard` `/hazard-game` | ゲーミフィケーション系の一貫性 |
| 低 | `/landing` `/contact` `/privacy` `/terms` `/forgot-password` | 静的・補助ページ |
| 低 | `/admin/*` | admin@test.com で管理画面の操作性 |

- 各ルートを**モバイル幅(390px)とデスクトップ幅(1280px)の両方**で確認する(実利用はスマホが主)
- **操作前後のスクリーンショットを保存**する。保存先: `docs/review/ux-audit-assets/<日付>/`。指摘には必ず該当スクショを紐づける — **証拠のない指摘は書かない**
- GPS・カメラ許可の拒否時挙動、データ0件の空状態も意図的に踏む

## Phase 2: 監査整理

以下7観点で指摘を整理する:

1. **導線・迷子**: 次のアクションが分かるか。行き止まり、戻れない画面、空状態の案内
2. **子ども向け適合**: ふりがな相当の配慮、文字量、タップ領域(44px以上)、アイコンだけで意味が通じるか
3. **フィードバック**: 操作後の成功/失敗の明示、ローディング中の無反応、エラーメッセージの分かりやすさ
4. **一貫性**: 「たんけんノート」デザイン(トークン=`lib/design/tanken.ts`)からの逸脱、ボタン・色・用語のブレ。危険度の色/ラベルは `lib/danger-level-presentation.ts` が唯一の正
5. **フォーム**: エラーの出方、必須項目の明示、GPS/カメラ拒否時のリカバリ
6. **パフォーマンス体感**: 待たされる場面のスケルトン/プログレス有無
7. **アクセシビリティ基礎**: コントラスト、フォーカス可視性、代替テキスト

### レポート形式(`docs/review/ux-audit-<日付>.md`)

指摘ごとに: `[重大度] ルート | 発生タスク | 現象 | ユーザーへの影響 | スクショパス`

- 重大度: **Critical**=タスク完遂不能 / **High**=多くのユーザーが挫折 / **Medium**=迷い・手戻り / **Low**=磨き込み
- 末尾に「重大度×修正コスト」マトリクスで**改善優先順位トップ10**を提示する

## Phase 3: 改善案の可視化

トップ10のうち**画面デザインで解決できるもの5〜8画面**を選び、それぞれ改善イメージを作る。

- **推奨手段: HTMLモックアップ + スクリーンショット**。`lib/design/tanken.ts` の実トークン(色・角丸・フォント)を使った自己完結HTMLを書き、Playwright(390px幅)で撮影する。実装可能性が担保され、Phase 4 でそのまま仕様になる
- 代替: 画像生成(Gemini等)。構図イメージの速いラフには可だが、実装乖離が出やすいので最終案はHTMLモックに寄せる
- 1画面につき: **現状スクショ + 改善案(可能なら2バリエーション) + 改善ポイント3点以内**
- 改善案は既存の「たんけんノート」デザイン言語(手書き風・温かい色・Zen Maru Gothic)を**維持**する。別物のデザインにしない

提示後、**必ず停止してユーザーのOKを待つ**。「どの案を採用するか」「修正したい点」を確認するまで Phase 4 に進まない。

## Phase 4: 実装指示書(承認後のみ)

- 採用された案ごとに、実装用の指示書を作成する: 対象ファイル、変更内容、完了条件(検証コマンド+実出力)、禁止手段(テストskip等)を含む「契約」形式
- 1タスク=1検証可能単位に分割する。実装前に characterization テストの要否を判断する
- 実装後は `verify-change` スキルの検証ゲートと `adversarial-review` を通す

## 禁止事項・制約(全フェーズ共通・厳守)

- **Phase 3 のOKが出るまでアプリのコードを変更しない**(成果物の md/HTML/画像ファイル追加のみ可)
- **`.env` `.env.local` 等の内容・環境変数の実値を読まない/出力しない**(キー名への言及は可)
- ファイル削除(`rm` 系)をしない。既存ファイルを上書きしない
- 操作は localhost:3000 のみ。本番・外部サービスへの送信をしない
- 報告フォーム等でDBに書き込む場合はタイトルに `[UXテスト]` を付けて識別可能にする
- サブエージェントに委譲する場合、上記の禁止事項をプロンプトに明記する(CLAUDE.md の削除事故の再発防止)

## 完了条件

- 全対象ルートを2画面幅で実操作した証拠(スクショ一式)がある
- 監査レポートに重大度つき指摘とトップ10が整理されている
- 5画面以上の改善案が「現状vs改善」対比で提示され、ユーザーの判断を待つ状態で停止している
